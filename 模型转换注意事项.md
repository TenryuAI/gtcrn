# GTCRN 模型转换注意事项

本文档总结了将 GTCRN PyTorch 流式模型导出为 ONNX 格式时遇到的两个关键错误，以及解决方案，供后续维护参考。

---

## 问题一：BatchNorm 融合逻辑错误

### 背景

PyTorch 2.9+ 的新 ONNX 导出器（dynamo 导出器）在导出含有 **分组卷积（groups > 1）** 的模型时，内部的 `onnxscript` 库会尝试自动融合 BatchNorm，但其实现存在 Bug，对 `groups > 1` 的卷积会产生形状广播错误：

```
ValueError: operands could not be broadcast together with shapes (16,8,1,5) (1,16,1,1)
```

为绕过此 Bug，采取了**预先手动融合 BN** 的策略（见 `fuse_bn.py`）。但预融合代码本身存在两处错误。

### 错误 1：对 ConvTranspose2d 错误调用了 Conv2d 融合函数

**根本原因**：`StreamGTConvBlock` 在 **解码器（decoder）** 中使用 `use_deconv=True`，此时 `point_conv1` 和 `point_conv2` 是 `nn.ConvTranspose2d` 类型，而非 `nn.Conv2d`。

**错误代码**（旧 `fuse_bn.py`）：
```python
# 错误：对所有 point_conv1 都调用 fuse_conv_bn，
# 但解码器中 point_conv1 是 ConvTranspose2d，
# fuse_conv_bn 返回的是 nn.Conv2d，完全改变了算子语义
child.point_conv1 = fuse_conv_bn(child.point_conv1, child.point_bn1)
```

**后果**：
- `fuse_conv_bn` 返回 `nn.Conv2d`，将转置卷积（上采样）替换成了普通卷积（下采样）
- 解码器的跳跃连接尺寸不匹配，导致模型崩溃或输出完全错误

**正确做法**：根据类型分别调用对应的融合函数：
```python
if isinstance(child.point_conv1, nn.ConvTranspose2d):
    child.point_conv1 = fuse_convtranspose_bn(child.point_conv1, child.point_bn1)
else:
    child.point_conv1 = fuse_conv_bn(child.point_conv1, child.point_bn1)
```

### 错误 2：StreamConv2d 的 BN 未被正确融合就被删除

**根本原因**：`StreamConv2d` 和 `StreamConvTranspose2d` 均继承自 `nn.Module`（而非 `nn.Conv2d`），内部卷积层以属性形式存储（`self.Conv2d` / `self.ConvTranspose2d`）。

**错误代码**（旧 `fuse_bn.py`）：
```python
# isinstance 判断为 False（StreamConv2d 不是 nn.Conv2d）
# hasattr(weight) 也为 False（weight 在 self.Conv2d 上，而非直接在 StreamConv2d 上）
# → 融合被跳过，但下面一行仍然执行！
if isinstance(child.depth_conv, nn.Conv2d):
    child.depth_conv = fuse_conv_bn(...)
else:
    if hasattr(child.depth_conv, 'weight'):  # ← 为 False，融合被跳过
        ...
child.depth_bn = nn.Identity()  # ← BN 被删除，但权重未融合进卷积！
```

**后果**：
- BN 层被替换为 `Identity`（相当于删除）
- 但 BN 的归一化效果（scale / shift）未被吸收进卷积权重
- 模型输出的数值分布完全改变，等价于模型权重错乱

**正确做法**：针对 `StreamConv2d` 和 `StreamConvTranspose2d` 访问其内部卷积层进行融合：
```python
dc = child.depth_conv
if isinstance(dc, nn.Conv2d):
    child.depth_conv = fuse_conv_bn(dc, child.depth_bn)
elif hasattr(dc, 'Conv2d') and isinstance(dc.Conv2d, nn.Conv2d):
    # StreamConv2d 的内部层
    dc.Conv2d = fuse_conv_bn(dc.Conv2d, child.depth_bn)
elif hasattr(dc, 'ConvTranspose2d') and isinstance(dc.ConvTranspose2d, nn.Conv2d):
    # StreamConvTranspose2d 的内部层（内部实际使用 nn.Conv2d 实现）
    dc.ConvTranspose2d = fuse_conv_bn(dc.ConvTranspose2d, child.depth_bn)
child.depth_bn = nn.Identity()
```

### 验证方法

融合完成后，务必验证融合前后模型输出的差异：

```python
import copy, torch
fused_model = copy.deepcopy(stream_model)
fuse_model(fused_model)

spec = torch.randn(1, 257, 1, 2)
cc = torch.zeros(2, 1, 16, 16, 33)
tc = torch.zeros(2, 3, 1, 1, 16)
ic = torch.zeros(2, 1, 33, 16)

with torch.no_grad():
    out1, _, _, _ = stream_model(spec, cc.clone(), tc.clone(), ic.clone())
    out2, _, _, _ = fused_model(spec, cc.clone(), tc.clone(), ic.clone())

diff = (out1 - out2).abs().max().item()
assert diff < 1e-4, f"BN 融合错误，差异过大：{diff}"
print(f"BN 融合验证通过，最大差异：{diff:.2e}")
```

> **注意**：调用时必须对缓存张量使用 `.clone()`，因为模型调用会对输入缓存进行**原地修改**（通过切片赋值 `conv_cache[0] = ...`），若不克隆则两次调用会使用不同的缓存状态，导致验证结果不可信。

## 问题二：新 ONNX 导出器无法正确导出 GRU 权重

### 背景

PyTorch 2.9+ 的新 dynamo 导出器（`torch.export.export` 路径）在导出包含 `nn.GRU` 的模型时，对 GRU 的内部权重 `_flat_weights` 处理不正确。导出时会出现如下警告：

```
UserWarning: The tensor attributes self.dpgrnn1.inter_rnn.rnn1._flat_weights[0], ...
were assigned during export. Such attributes must be registered as buffers
using the `register_buffer` API.
```

### 后果

GRU 权重未能正确保存到 ONNX 文件中，导致 ONNX 模型推理时使用了**错误的权重**（可能是零或随机值）。最终表现为：

- ONNX 模型输出与 PyTorch 模型输出相差巨大（实测差异约 1.5，正常应 < 1e-4）
- C++ 程序降噪效果极差，与 Python 版本完全不同

### 解决方案

在 `torch.onnx.export` 调用时，添加 `dynamo=False` 参数，强制使用**旧版 TorchScript 追踪（JIT）导出器**：

```python
torch.onnx.export(
    stream_model,
    (spec, conv_cache, tra_cache, inter_cache),
    onnx_path,
    input_names=['mix', 'conv_cache', 'tra_cache', 'inter_cache'],
    output_names=['enh', 'conv_cache_out', 'tra_cache_out', 'inter_cache_out'],
    opset_version=17,
    dynamo=False,  # 关键：使用旧版 JIT 导出器，正确处理 GRU 权重
    verbose=False
)
```

旧版 JIT 导出器通过 TorchScript 追踪捕获计算图，能够将 GRU 的 `_flat_weights` 正确地转换为 ONNX 的 GRU 算子常量权重。

### 导出后的验证方法

使用 ONNX Runtime 运行导出的模型并与 PyTorch 对比（见 `validate_cpp.py`）：

```python
import onnxruntime, numpy as np, torch

session = onnxruntime.InferenceSession('onnx_models/gtcrn_simple.onnx')
spec_pt = torch.randn(1, 257, 1, 2)

with torch.no_grad():
    out_pt, _, _, _ = stream_model(spec_pt, 
        torch.zeros(2,1,16,16,33), torch.zeros(2,3,1,1,16), torch.zeros(2,1,33,16))

out_ort = session.run(None, {
    'mix': spec_pt.numpy(),
    'conv_cache': np.zeros([2,1,16,16,33], dtype=np.float32),
    'tra_cache': np.zeros([2,3,1,1,16], dtype=np.float32),
    'inter_cache': np.zeros([2,1,33,16], dtype=np.float32)
})

diff = np.abs(out_pt.numpy() - out_ort[0]).max()
assert diff < 1e-3, f"ONNX 导出错误，差异过大：{diff}"
print(f"ONNX 验证通过，最大差异：{diff:.2e}")
```

## 附：IR Version 兼容性问题

PyTorch 新版导出的 ONNX 模型 IR version 为 10，而 C++ 侧使用的 ONNX Runtime 1.17.1 最高只支持 IR version 9。

**解决方案**：导出后手动将 IR version 降级：

```python
import onnx
m = onnx.load('onnx_models/gtcrn_simple.onnx')
m.ir_version = 8  # 降级到与 ONNX Runtime 1.17.1 兼容的版本
onnx.save(m, 'onnx_models/gtcrn_simple.onnx')
```

> 此操作仅修改文件头部的版本号，不影响模型权重和算子。

## 完整导出流程（export_onnx.py）

```
1. 加载 PyTorch 权重 → GTCRN 模型
2. 转换为流式模型 → StreamGTCRN (convert_to_stream)
3. 手动预融合 BN   → fuse_model(stream_model)  [见 fuse_bn.py]
4. 验证 BN 融合正确性（使用 .clone() 克隆缓存）
5. ONNX 导出       → dynamo=False, opset_version=17
6. ONNX 模型检查   → onnx.checker.check_model()
7. ONNX 简化       → onnxsim.simplify()
8. IR Version 降级 → m.ir_version = 8
9. 验证 ONNX 输出  → 与 PyTorch 模型对比（差异应 < 1e-3）
```
